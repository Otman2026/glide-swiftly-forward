# خطة الإصلاح الشاملة — SAIFO TRANSPORT

## 1) قفل حساب مالك النظام (System Owner)

- إضافة **مستخدم واحد فقط** ثابت في قاعدة البيانات لبريد `otnaj.2017@gmail.com` مع دور `system_owner`.
- إنشاء تريغرات على `user_roles` تمنع:
  - إسناد `system_owner` لأي مستخدم آخر (منع INSERT).
  - نسخه أو ترقيته (منع UPDATE للدور `system_owner`).
  - حذفه (منع DELETE لسجل مالك النظام الوحيد).
- إخفاء زر «إسناد دور system_owner» من واجهة المستخدمين.
- ضبط RLS حصرية للوحة المالك: قراءة كل الشركات والاشتراكات فقط لحامل هذا الدور.
- إنشاء الحساب أولاً في Supabase Auth بكلمة مرور مؤقتة، ثم يمكن للمالك تغييرها لاحقاً.

## 2) فصل Username عن Email

- إضافة عمود `username` (اختياري، unique إذا وُجد) في `profiles`.
- تعديل صفحة التسجيل والدخول:
  - الدخول متاح بالبريد **أو** اسم المستخدم.
  - عند التسجيل: الاسم الكامل + username اختياري + email (للاسترجاع فقط) + كلمة مرور.
- إضافة صفحة «حسابي» لتحديث username لاحقاً.
- منطق تحويل: إذا كان المُدخل ليس بريدًا، ابحث في `profiles.username` → استخرج البريد الحقيقي → سجّل الدخول به.

## 3) وحدة التراخيص والاشتراكات (لوحة المالك فقط)

توحيد كل العمليات داخل `/app/system-owner` مع تبويبات:

**أ. الشركات (Tenants)**
- عرض/تعديل/تعليق (`suspended`) / إعادة تفعيل.

**ب. الاشتراكات**
- إنشاء اشتراك يدوي لأي شركة.
- تمديد (Extend +30/+90/+365 يوم).
- إيقاف (`status=cancelled`) / إعادة تفعيل.
- تغيير الخطة يدوياً.

**ج. Trial**
- بدء/تمديد/إنهاء فترة تجريبية.

**د. License Keys** (جدول جديد `license_keys`)
- إنشاء مفتاح ترخيص (تلقائي بصيغة `SAIFO-XXXX-XXXX-XXXX-XXXX`).
- ربط بالشركة والخطة.
- تعديل/إلغاء (`revoked_at`).
- تاريخ انتهاء + عدد الاستخدامات.

**هـ. الخطط والأسعار** (جدول جديد `plans`)
- CRUD كامل: اسم، سعر شهري، سعر سنوي، حد المستخدمين، حد المركبات، مزايا.
- استخدامها في صفحة `/app/billing` بدل الثابت الحالي.

**و. طلبات الفوترة** (موجودة — تحسين العرض فقط).

## 4) فحص وإصلاح الوحدات

مراجعة سريعة لكل صفحات `/app/*` والتأكد من:
- كل زر مرتبط بمنطق (لا أزرار «TODO»).
- كل نموذج يحفظ في قاعدة البيانات.
- كل عرض يقرأ من الجدول الصحيح.

سيتم توثيق الفروقات في التقرير النهائي.

## 5) التقرير النهائي

Markdown في `/app/system-owner` تحت تبويب «تقرير الجاهزية» يعرض:
- ✅ ما تم إصلاحه
- ⚠️ ما يحتاج إعداد يدوي (مثل تفعيل Stripe)
- 📋 قائمة الوحدات وحالتها

## التنفيذ على 3 دفعات

**دفعة A — قاعدة البيانات (Migration واحد):**
1. جدول `plans` + بذور 4 خطط.
2. جدول `license_keys` + RLS + trigger لتوليد المفتاح.
3. تريغرات حماية `system_owner` (منع INSERT/UPDATE/DELETE من أي طرف).
4. عمود `profiles.username` (unique, nullable).
5. إنشاء المستخدم `otnaj.2017@gmail.com` + دور `system_owner` (عبر `supabase.auth.admin` في دالة بذر منفصلة).
6. GRANTs الكاملة.

**دفعة B — الواجهات:**
1. `app.system-owner.tsx`: إعادة هيكلة كاملة بتبويبات (شركات/اشتراكات/تراخيص/خطط/طلبات/تقرير).
2. `auth.tsx`: دعم username + email.
3. `app.users.tsx`: إخفاء دور system_owner + عرض username.
4. `app.billing.tsx`: قراءة الخطط من جدول `plans`.

**دفعة C — التحقق والتقرير:**
- فحص كل صفحة، توليد التقرير النهائي داخل النظام.

## ملاحظات تقنية

- **لا يمكن إنشاء مستخدم Auth عبر SQL migration مباشرة** بشكل موثوق. لذا الحل:
  - Migration ينشئ الجدول والتريغرات فقط.
  - عند أول تحميل، إذا لم يوجد `system_owner`، ندعو دالة server function مع service role تُنشئ الحساب `otnaj.2017@gmail.com` بكلمة مرور مؤقتة `SaifoOwner2026!` (سيُطلب من المالك تغييرها فور الدخول).
- الحماية النهائية على قاعدة البيانات: التريغر يرفض أي محاولة إضافة `system_owner` ثانٍ حتى لو تم عبر SQL مباشر.
- في `has_role`، `system_owner` تعطي وصولاً كاملاً لكل الجداول عبر السياسات الحالية.

هل أنطلق بهذه الخطة؟ ستستغرق 3 ردود متتالية للتنفيذ الكامل.
